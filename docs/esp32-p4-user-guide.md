# Руководство: ESP32-P4 FPV стример

> **Статус реализации:**
> - ✅ PROMPT-01 — структура проекта, скелет, сборка проходит
> - 🔲 PROMPT-02 — WiFi AP + HTTP/WebSocket сигналинг
> - 🔲 PROMPT-03 — WebRTC стек (libpeer)
> - 🔲 PROMPT-04 — Камера OV5647 + H.264 энкодер
> - 🔲 PROMPT-05 — DataChannel (ping/pong, ts-метки)
> - 🔲 PROMPT-06 — Интеграция, сквозной тест
>
> До завершения PROMPT-02..06 прошивка запускается, но видео не передаёт.
> Инструкции ниже актуальны для полностью реализованной прошивки.

Инструкция по подключению ESP32-P4-Function-EV-Board с камерой OV5647 в качестве
полностью автономного FPV-стримера для Quest 2.

**Ноутбук нужен только один раз — для прошивки.**
После этого плата работает самостоятельно: поднимает WiFi-сеть "FPV-Drone", сама
раздаёт веб-страницу и управляет WebRTC-соединением. Quest 2 подключается напрямую.

---

## Что понадобится

### Оборудование

| Компонент | Примечание |
|-----------|-----------|
| ESP32-P4-Function-EV-Board | Официальная плата Espressif |
| OV5647 Camera Module | Raspberry Pi Camera Module v1/v2 (5MP, MIPI CSI) |
| USB-C кабель | Для прошивки (один раз) и питания в полёте |
| Ноутбук (macOS/Linux) | Только для сборки и прошивки |
| Quest 2 | Подключается к WiFi-сети ESP32 |

### Программное обеспечение

| ПО | Версия | Для чего |
|----|--------|---------|
| ESP-IDF | **5.4 или новее** | Сборка прошивки (5.3 не поддерживает ESP32-P4 v1.x) |
| Python | 3.8+ | Инструменты ESP-IDF |
| Git | Любая | Клонирование репозитория |

> Node.js и `npm start` **не нужны** — сервер встроен в прошивку.

## Архитектура системы

```
OV5647 Camera
     │ MIPI CSI
     ▼
ESP32-P4-Function-EV-Board
  ├─ WiFi AP "FPV-Drone" (192.168.4.1)
  ├─ HTTP-сервер :8080 → index.html, JS, CSS
  ├─ WebSocket /ws → сигналинг SDP/ICE
  ├─ H.264 VEU (аппаратный кодировщик)
  └─ WebRTC peer connection
          │ UDP прямо в той же сети
          ▼
      Quest 2 (192.168.4.2)
  ├─ WiFi → "FPV-Drone"
  ├─ WebRTC декодер (MediaCodec H.264)
  ├─ OpenXR стерео-рендеринг
  └─ DataChannel (E2E статистика)
```

Сигналинг (обмен SDP/ICE) и видеопоток — всё в одной WiFi-сети, напрямую между
ESP32 и Quest. Никаких промежуточных серверов нет.

## Шаг 1 — Установка ESP-IDF

> Если ESP-IDF уже установлен и `idf.py --version` показывает 5.3+, пропустите шаг.

### macOS

```bash
# Зависимости
brew install cmake ninja dfu-util

# Скачать ESP-IDF v5.3
mkdir -p ~/esp && cd ~/esp
git clone --recursive --branch v5.4.1 https://github.com/espressif/esp-idf.git
cd esp-idf

# Установить цепочку инструментов для ESP32-P4
./install.sh esp32p4

# Добавить в PATH текущей сессии
. ~/esp/esp-idf/export.sh
```

**Проверка:**
```bash
idf.py --version
# Ожидаемый вывод: ESP-IDF v5.3.x
```

> **Совет:** добавьте в `~/.zshrc`:
> ```bash
> alias get_idf='. ~/esp/esp-idf/export.sh'
> ```
> Затем вместо длинной команды просто: `get_idf`

## Шаг 2 — Подключение камеры OV5647

OV5647 подключается к разъёму **MIPI CSI** через ленточный шлейф (FPC).

```
ESP32-P4-Function-EV-Board
┌─────────────────────────────────┐
│                                 │
│  [MIPI CSI разъём]              │
│   ┌──────────────┐              │
│   │  FPC кабель  │◄── OV5647   │
│   └──────────────┘              │
│                                 │
│  [USB-C]◄── кабель к ноутбуку  │
│             (прошивка / питание)│
└─────────────────────────────────┘
```

**Важно:**
- Подключайте/отключайте камеру только при отключённом питании
- Шлейф вставляется до щелчка фиксатора (защёлкнуть замок коннектора)
- Металлическая сторона контактов направлена в сторону, указанную маркировкой на плате

## Шаг 3 — Настройка прошивки

Откройте `esp32-p4-streamer/main/config.h`. В новой архитектуре **не нужно указывать
IP-адреса или параметры чужой WiFi-сети** — ESP32 сама создаёт точку доступа.

При желании измените SSID или добавьте пароль:

```c
// WiFi Access Point — ESP32-P4 создаёт собственную сеть
#define WIFI_AP_SSID    "FPV-Drone"   // имя вашей сети (можно изменить)
#define WIFI_AP_PASS    ""            // пусто = открытая сеть
                                      // или: "MyPassword" = WPA2

// Порт HTTP/WebSocket-сервера (менять не нужно)
#define SERVER_PORT     8080
```

> После включения ESP32 сеть "FPV-Drone" появится в WiFi-списке Quest 2
> через ~5 секунд. IP-адрес ESP32 всегда `192.168.4.1`.

## Шаг 4 — Сборка и прошивка

### 4.1 Подключить ESP32-P4 к ноутбуку

```bash
ls /dev/cu.usb*
# Пример: /dev/cu.usbmodem14301
```

Если устройство не появилось — нужен кабель с поддержкой данных (не только зарядный).

### 4.2 Активировать ESP-IDF

```bash
. ~/esp/esp-idf/export.sh
# или: get_idf
```

### 4.3 Собрать прошивку

```bash
cd esp32-p4-streamer
idf.py set-target esp32p4
idf.py build
```

Первая сборка: 3–5 минут. Последующие — значительно быстрее.

### 4.4 Прошить (firmware + веб-файлы)

```bash
idf.py -p /dev/cu.usbmodem14301 flash monitor
```

Команда одновременно прошивает:
- Основную прошивку (fpv_streamer.elf)
- SPIFFS-раздел с веб-файлами из `public/` (index.html, JS, CSS)

**Ожидаемый вывод в мониторе:**
```
I (xxx) main: FPV ESP32-P4 Streamer starting...
I (xxx) signaling: SPIFFS mounted at /www
I (xxx) signaling: WiFi AP started: SSID=FPV-Drone, IP=192.168.4.1, port=8080
I (xxx) camera: OV5647 initialized 1280x720@30fps
I (xxx) camera: First H.264 frame: 28214 bytes, encode=9ms
```

> Для выхода из монитора: **Ctrl + ]**

После этого ноутбук можно отключить. ESP32 готова к работе автономно.

## Шаг 5 — Запуск трансляции

### 5.1 Подать питание на ESP32-P4

Подключить USB-C к любому источнику питания 5V (powerbank, зарядник).
Через ~5 секунд сеть "FPV-Drone" появится в WiFi-списке устройств.

### 5.2 Подключить Quest 2 к WiFi "FPV-Drone"

На Quest 2: **Settings → Wi-Fi → FPV-Drone** → Connect.

> Если сеть открытая (`WIFI_AP_PASS = ""`), пароль не запрашивается.
> Quest 2 предупредит "No internet" — это нормально, нажать **Connect Anyway**.

### 5.3 Открыть видео

**Вариант A — Нативное приложение** (рекомендуется, минимальная задержка):
- Запустить FPV Quest на Quest 2
- Нажать кнопку **Y** → откроется VR-панель ввода адреса
- Левым стиком установить IP: **192.168.4.1**, порт: **8080**
- Нажать **A** → Connect
- WS-адрес: `ws://192.168.4.1:8080/ws`

**Вариант B — Браузер** (для разовой проверки):
- Открыть Oculus Browser
- Перейти: `http://192.168.4.1/`

### 5.4 Что должно произойти

В мониторе ESP32 (если подключён):
```
I (xxx) signaling: Viewer WebSocket connected
I (xxx) signaling: Received role=viewer
I (xxx) webrtc: Creating offer...
I (xxx) webrtc: ICE connected (host candidates, 192.168.4.x)
I (xxx) main: WebRTC connected — streaming
I (xxx) datachannel: Channel open
```

Через 2–5 секунд: видео с OV5647 в VR. ✓

## Статистика задержки

После ~4–5 секунд (синхронизация DataChannel) в HUD Quest 2:

| Метрика | Значение | Что означает |
|---------|----------|-------------|
| E2E | **20–50 мс** | Полная задержка: камера → WiFi → декод → дисплей |
| Encode | 5–15 мс | Аппаратный H.264 VEU на ESP32-P4 |
| Net | 5–15 мс | RTT/2 (прямая связь, нет роутера) |

> E2E в AP-режиме ниже, чем через роутер (~65–100 мс), т.к. нет лишних сетевых хопов.
> Если `Encode > 0` — аппаратный кодировщик работает ✓

## Диагностика проблем

### Сеть "FPV-Drone" не появляется в WiFi
Плата ещё загружается или не подала питание.
Подождите 10–15 секунд, при необходимости перезагрузите (кнопка Reset на плате).

---

### Quest 2 не подключается к "FPV-Drone"
- Убедитесь, что нажали **Connect Anyway** при предупреждении "No internet"
- Если задан пароль в `config.h` — проверьте его корректность

---

### WebSocket не подключается (нативное приложение)
Проверьте, что введён правильный адрес в VR-панели:
- IP: `192.168.4.1` (не 192.168.1.x или другие)
- Порт: `8080`
- Путь в SignalingClient.kt: `/ws`

---

### ICE connection failed — нет видео
В AP-режиме STUN не используется; ICE соединяется через host candidates.
Если ICE не проходит:
- Оба устройства должны быть в сети **"FPV-Drone"** (не в другой WiFi)
- Проверьте, что Quest 2 получил IP 192.168.4.x (Settings → Wi-Fi → "FPV-Drone")

---

### Камера не инициализируется
```
E (xxx) camera: esp_camera_init failed: ESP_ERR_NOT_FOUND
```
Переподключите шлейф камеры при отключённом питании.

---

### SPIFFS ошибка при монтировании
```
E (xxx) signaling: SPIFFS mount failed
```
Веб-файлы не были записаны. Перепрошейте с ноутбука:
```bash
idf.py -p /dev/cu.usbmodem* flash
```

---

### Видео есть, HUD stats пустой первые 5 секунд
Нормально — DataChannel синхронизирует часы (5 round-trips).
Если через 15+ секунд stats не появились — проверьте лог ESP32 на ошибки DataChannel.
